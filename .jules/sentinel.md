## 2026-03-18 - Prevent Mobile Offline Time-Tampering
**Vulnerability:** The mobile app calculated offline punch timestamps dynamically by computing the delay between the local device time of the action and the local device time during a future sync. A malicious user could disable their network connection, change the device clock to a past time, log a fake check-in, and reset the clock before reconnecting. The backend would falsely register the forged historical timestamp.
**Learning:** Purely local timestamps and delay calculations cannot be trusted on client devices, particularly for offline-first architecture. A device's local clock is fully mutable by the user and does not represent an objective progression of time.
**Prevention:** Establish a trusted monotonic baseline while online. Fetch an authoritative timestamp from the server, calculate an offset against the local device time, and persist this offset. For all subsequent offline events, compute the real time by applying the stored offset to the current local device time. Furthermore, track the `lastLocalSyncTime` to instantly detect and reject actions if the local clock jumps backward (which indicates tampering). This guarantees the backend always receives an accurate, independently verified timeline.
