## 2026-03-18 - Prevent Mobile Offline Time-Tampering
**Vulnerability:** The mobile app calculated offline punch timestamps dynamically by computing the delay between the local device time of the action and the local device time during a future sync. A malicious user could disable their network connection, change the device clock to a past time, log a fake check-in, and reset the clock before reconnecting. The backend would falsely register the forged historical timestamp.
**Learning:** Purely local timestamps and delay calculations cannot be trusted on client devices, particularly for offline-first architecture. A device's local clock is fully mutable by the user and does not represent an objective progression of time.
**Prevention:** Establish a trusted monotonic baseline while online. Fetch an authoritative timestamp from the server, calculate an offset against the local device time, and persist this offset. For all subsequent offline events, compute the real time by applying the stored offset to the current local device time. Furthermore, track the `lastLocalSyncTime` to instantly detect and reject actions if the local clock jumps backward (which indicates tampering). This guarantees the backend always receives an accurate, independently verified timeline.
## 2026-03-18 - Prevent UI Time Tampering via Shadow Tick
**Vulnerability:** A previous fix introduced a `Date.now() + serverOffset` mechanism to calculate "True Time". However, relying on `Date.now()` within the UI state intervals caused the UI's progress bar to jump forwards or backward instantly if the user actively changed their phone's clock while the app was running, providing a false perception of success.
**Learning:** If an interval logic evaluates `Date.now()`, any underlying OS clock manipulation will instantly cascade into the React render cycle, exposing the "True Time" mechanic to the local clock's manipulation.
**Prevention:** Decouple intervals completely from `Date.now()`. Initialize a `shadowTimeRef` at component mount and strictly manually increment it by `+1000` ms inside the interval loop. Simultaneously, calculate the expected OS drift (`Date.now() + serverOffset` vs `shadowTimeRef.current`). If drift exceeds a predefined threshold (e.g., 60,000 ms) because the user actively changed their OS clock, immediately trigger a blocking UI state. This establishes a "Shadow Clock" that cannot be manipulated via system settings.
## 2026-03-18 - Prevent UI Permanent Block (Sleep Debt)
**Vulnerability:** The UI used a `shadowTimeRef` incrementing precisely by 1000ms every interval. When the mobile app was suspended or put into the background, the JS interval paused. Upon waking up, the `shadowTimeRef` was severely delayed ("Sleep Debt"). The time-tamper check incorrectly interpreted this massive drift (between the delayed `shadowTimeRef` and actual elapsed OS time) as a malicious user tampering with the device clock, permanently disabling the app's punch features until restart.
**Learning:** Pure `setInterval` logic pauses during backgrounding in React Native environments. "Shadow Clocks" that rely on manual incrementation must explicitly handle lifecycle sleep debt.
**Prevention:** Inside the component's `useEffect`, right before the `setInterval` begins, actively synchronize `shadowTimeRef.current = Date.now() + serverTimeOffset`. Because this dependency array triggers on app re-focus (when `serverTimeOffset` refreshes via `AppState` sync in the store), the shadow clock instantly "fast-forwards" to the correct real-world time, completely eliminating the false-positive drift alert and successfully self-healing the state.
