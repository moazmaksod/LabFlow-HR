## 2026-03-18 - Prevent Mobile Offline Time-Tampering
**Vulnerability:** The mobile app calculated offline punch timestamps dynamically by computing the delay between the local device time of the action and the local device time during a future sync. A malicious user could disable their network connection, change the device clock to a past time, log a fake check-in, and reset the clock before reconnecting. The backend would falsely register the forged historical timestamp.
**Learning:** Purely local timestamps and delay calculations cannot be trusted on client devices, particularly for offline-first architecture. A device's local clock is fully mutable by the user and does not represent an objective progression of time.
**Prevention:** Establish a trusted monotonic baseline while online. Fetch an authoritative timestamp from the server, calculate an offset against the local device time, and persist this offset. For all subsequent offline events, compute the real time by applying the stored offset to the current local device time. Furthermore, track the `lastLocalSyncTime` to instantly detect and reject actions if the local clock jumps backward (which indicates tampering). This guarantees the backend always receives an accurate, independently verified timeline.

## 2026-03-18 - Prevent UI Time Tampering via Shadow Tick
**Vulnerability:** A previous fix introduced a `Date.now() + serverOffset` mechanism to calculate "True Time". However, relying on `Date.now()` within the UI state intervals caused the UI's progress bar to jump forwards or backward instantly if the user actively changed their phone's clock while the app was running, providing a false perception of success.
**Learning:** If an interval logic evaluates `Date.now()`, any underlying OS clock manipulation will instantly cascade into the React render cycle, exposing the "True Time" mechanic to the local clock's manipulation.
**Prevention:** Decouple intervals completely from `Date.now()`. Initialize a `shadowTimeRef` at component mount and strictly manually increment it by `+1000` ms inside the interval loop. Simultaneously, calculate the expected OS drift (`Date.now() + serverOffset` vs `shadowTimeRef.current`). If drift exceeds a predefined threshold (e.g., 60,000 ms) because the user actively changed their OS clock, immediately trigger a blocking UI state. This establishes a "Shadow Clock" that cannot be manipulated via system settings.

## 2026-03-18 - Prevent UI Permanent Block (Sleep Debt)
**Vulnerability:** The UI used a `shadowTimeRef` incrementing precisely by 1000ms every interval. When the mobile app was suspended or put into the background, the JS interval paused. Upon waking up, the `shadowTimeRef` was severely delayed ("Sleep Debt"). The time-tamper check incorrectly interpreted this massive drift (between the delayed `shadowTimeRef` and actual elapsed OS time) as a malicious user tampering with the device clock, permanently disabling the app's punch features until restart.
**Learning:** Pure `setInterval` logic pauses during backgrounding in React Native environments. "Shadow Clocks" that rely on manual incrementation must explicitly handle lifecycle sleep debt.
**Prevention:** Inside the component's `useEffect`, right before the `setInterval` begins, actively synchronize `shadowTimeRef.current = Date.now() + serverTimeOffset`. Because this dependency array triggers on app re-focus (when `serverTimeOffset` refreshes via `AppState` sync in the store), the shadow clock instantly "fast-forwards" to the correct real-world time, completely eliminating the false-positive drift alert and successfully self-healing the state.

## 2026-03-29 - Secure Randomness for Unique Identifiers
**Vulnerability:** The system used `Math.random()` to generate suffixes for file uploads and device IDs. This is a PRNG (Pseudo-Random Number Generator) that is cryptographically insecure, meaning the sequences it produces can be predicted if the internal state is known or guessed. This could lead to filename collisions, enumeration attacks, or predictable device identifiers.
**Learning:** `Math.random()` should never be used for security-sensitive values like unique IDs, tokens, or filenames where unpredictability is required.
**Prevention:** Use cryptographically secure alternatives: `crypto.randomBytes()` in Node.js environments and `window.crypto.getRandomValues()` in browser environments. These APIs utilize system-level entropy to generate truly unpredictable random values.

## 2024-03-29 - Missing File Upload Validation (MIME Type)
**Vulnerability:** Arbitrary file upload due to missing MIME type and file size validation on the `/upload-avatar` endpoint, potentially leading to XSS, RCE, or DoS.
**Learning:** Security validations for Multer should be centralized and reused across all file upload routes to prevent drift and ensure consistent security posture. Old files must also be explicitly deleted to prevent resource exhaustion.
**Prevention:** Created a shared `fileUpload.ts` utility defining strict `fileFilter` (allowing only image/jpeg, image/png, image/webp) and `limits` (5MB). Applied this to both user and settings routes, ensuring any Multer error is caught and returns a 400 Bad Request instead of a generic 500 error.

## 2026-04-12 - Stale JWT Role Allows Privilege Persistence After Role Revocation
**Vulnerability:** `authMiddleware.ts` verified the JWT signature and confirmed the user still existed in the DB, but then set `req.user = decoded` — using the **role from the token**, not from the database. When a manager revoked an employee's role (demoting to `pending`, or the system suspended them), the change had zero effect until the 7-day token expired. A terminated or demoted employee retained full `employee`-level API access (clock-in, requests, payroll records) for up to a week.
**Learning:** The "user exists" DB check gave a false sense of security. It prevented use of tokens belonging to deleted accounts, but said nothing about whether the user's *current role* matched their token's role claim. The JWT was trusted as the authoritative source of role — but JWTs are immutable after issuance. Role is a mutable property that lives in the database, not in the token.
**Prevention:** Changed `SELECT id FROM users` to `SELECT id, role FROM users` in the authenticate middleware, then set `req.user = { id: user.id, role: user.role }` using the **fresh DB value**. The token still performs its job (cryptographic integrity + expiry), while the DB provides the current authoritative role on every request. Zero performance cost since the query was already running.

## 2026-04-11 - No Rate Limiting on Authentication Endpoints
**Vulnerability:** The `/api/auth/login` and `/api/auth/register` endpoints had no rate limiting. An attacker could make unlimited password guesses against any known email address (brute force) or flood the system with fake account registrations (account spam). The `bcrypt.compare()` call in login takes ~100ms per attempt, meaning even with bcrypt's intentional slowness, an attacker can test ~600 passwords/minute per IP.
**Learning:** Authentication endpoints are the most targeted attack surface in any web application. Even with bcrypt hashing, no rate limiting means an attacker with a common password list (rockyou.txt has 14M entries) could eventually compromise weak passwords. This is especially dangerous for an HR system with sensitive payroll/attendance data. The absence of this protection was not obvious from code inspection alone — it's easy to miss a missing safeguard vs. a present vulnerability.
**Prevention:** Added an in-memory sliding window rate limiter (`server/middlewares/rateLimitMiddleware.ts`) with no external dependencies: 10 login attempts per 15 minutes and 5 registrations per hour per IP. Applied before the route handler in `authRoutes.ts`. Returns HTTP 429 with a `Retry-After` header. Includes a cleanup interval to prevent memory leaks from unbounded Map growth.
